# Problem Intelligence Platform
## Project Overview for New Team Members

## 1) What We Are Building
We are building a **Problem Intelligence Platform**: a web application that collects real-world problems from multiple sources, structures them into a clear format, analyzes them with AI, and presents them in a feed-like interface so people can discover startup opportunities more easily.

The product is designed to help users answer a very important question:

**What problem is worth solving?**

Instead of starting with random startup ideas, the platform starts with real pain points from the world and turns them into structured opportunity intelligence.

## 2) Why This Product Exists
Many people want to build something meaningful, but they do not know where to begin. They often ask:
- What should I build?
- Which problems are real?
- Which problems are important enough to solve?
- Which problems are big enough to become a business?

These questions are difficult because real-world problems are scattered across forums, reports, research papers, communities, and government sources. They are not organized in a way that makes them easy to compare or understand.

This platform exists to solve that discovery problem.

## 3) Core Idea
The platform converts scattered signals into **Problem Intelligence Cards**.

A problem card is a structured record that includes:
- the problem statement
- the sector
- the user group affected
- the geography
- frequency / repetition
- pain point analysis
- possible solution directions
- scoring
- source signals
- confidence level

Each card is meant to be easy to scan quickly, but also deep enough for serious evaluation.

### Supported Sectors
The platform organizes problems into these industry verticals:
- Healthcare
- Fintech
- Education
- Agriculture
- GovTech / Legal
- CleanTech
- Employment / EdTech
- Creator Economy
- Retail
- Rare Disease
- Technology

## 4) Product Vision
The long-term vision is to create a system that works like a **news app for real-world problems**, but instead of entertainment or general news, it helps users discover opportunities for innovation.

The platform should eventually help users:
- find painful, recurring problems
- understand which problems matter most
- compare opportunities across sectors
- identify AI-solvable and startup-worthy gaps
- turn observations into founder-level insight

## 5) Who This Product Is For
The primary users are:
- aspiring entrepreneurs
- student founders
- indie builders
- AI product creators
- startup enthusiasts

Secondary users may include:
- founders
- researchers
- analysts
- innovation teams
- investors looking for problem-led opportunity discovery

## 6) What Makes This Different
This is **not** a standard directory, **not** a generic dashboard, and **not** just a scraping tool.

The difference is the intelligence layer.

The platform does not only collect problems; it:
- classifies them by sector and geography
- estimates severity and opportunity
- identifies pain points and root causes
- suggests possible solution directions
- highlights how trustworthy the signal is

This is what turns raw information into decision support.

## 7) Product Strategy
The product is **India-first** and **geo-aware**.

That means:
- the MVP focuses on India
- every problem is tagged with geography
- the system is designed so it can expand globally later
- scoring and context should always respect the location of the problem

India is the starting point because it provides strong real-world relevance, easier validation, and clear context.

## 8) High-Level User Experience
The user experience is based on two main layers:

### Layer 1: Feed / Discovery
Users browse problems quickly in a news-style feed.
They can swipe, scroll, or tap through problem cards.
This layer is designed for quick scanning and rapid discovery.

### Layer 2: Deep Dive / Analysis
When a user taps a problem, they see the full breakdown.
This includes the problem itself, pain points, scoring, source signals, and possible solutions.
This layer is designed for evaluation and decision-making.

## 9) Core Workflow of the Platform
The platform generally follows this flow:

1. Problems are collected from source platforms
2. The system cleans and standardizes the data
3. AI agents classify and analyze each problem
4. The problem is scored and enriched with insights
5. The UI displays it as a problem card
6. The user scans the feed and opens details if interested

## 10) Main Building Blocks
The overall project has five major parts:

### A. Product and UI
The front-end experience, including the feed, detail page, filters, responsive layout, and interaction design.

### B. Problem Data Model
The schema that defines how every problem is stored and represented.

### C. AI Agent System
Specialized agents that help collect, classify, summarize, score, and interpret problems.

### D. Backend and Ingestion Flow
The logic that brings in data, processes it, stores it, and makes it available for the UI.

### E. Scoring and Ranking Engine
The logic that determines which problems are more severe, more valuable, and more promising.

## 11) Why AI Is Important Here
AI is not the product by itself.
AI is the analysis engine.

Its role is to help with:
- source extraction
- classification
- insight generation
- scoring
- solution brainstorming
- trend detection

Without AI, the platform would just be a static list.
With AI, it becomes an intelligence system.

## 12) Why the Project Needs Structure
This project will only work if the team treats problems like data objects with a consistent format.

That is why the work must be organized into:
- project overview
- UI brief
- problem schema
- agent architecture
- backend flow
- scoring logic
- code generation spec

Each file has a specific purpose. This prevents confusion and makes the system scalable.

## 13) The Importance of the Problem Card
The problem card is the core unit of value in the entire product.

Every card should answer:
- What is the issue?
- Who is affected?
- How serious is it?
- Why does it matter?
- What could solve it?
- How strong is the evidence?

If a card does not help answer these questions, it is not useful enough for the platform.

## 14) Design Philosophy
The interface should feel:
- clean
- modern
- structured
- trustworthy
- intelligent
- easy to scan

It should avoid unnecessary visual noise and always prioritize clarity.

The product should feel serious enough for founders and researchers, but still simple enough for a student to use quickly.

## 15) Main Product Outcome
The goal is not just to show problems.
The goal is to help users **notice opportunities hidden inside problems**.

That means the platform should help people think like founders:
- identify pain
- understand context
- estimate value
- imagine solutions
- prioritize what is worth building

## 16) What Success Looks Like
The project succeeds when a user can open the app and immediately say:
- this problem feels real
- this problem is important
- this problem is worth exploring
- I can imagine building something here

That is the outcome we are designing for.
